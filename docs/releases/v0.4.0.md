## Highlights

RDAPify 0.4.0 is a major security and architecture release. The public API is unchanged — all existing code continues to work without modification.

---

## Security Improvements

### SSRF Protection Hardening

- **IPv4-mapped IPv6 bypass fixed** — addresses like `::ffff:192.168.1.1` were not blocked in case-insensitive matching. Fixed in both `rdapify` (npm) and `rdap-security` (Rust).
- **CGN range blocked** — RFC 6598 Carrier-Grade NAT range (`100.64.0.0/10`) added to the private range blocklist.
- **DNS pre-resolution** — the `rdap-security` crate now resolves hostnames before connecting and validates the resolved IP against the private range list, preventing DNS rebinding attacks.
- **Redirect chain guard** — redirect chains are validated at each hop. Redirects to private IPs are blocked.
- **Response size limits** — RDAP JSON responses are now capped at a configurable limit before deserialization.

### RDAP Response Validation

The `rdap-core` crate adds a JSON validation layer that:
- Enforces maximum response size (configurable via `rdapify.toml`)
- Validates content-type before deserialization
- Rejects malformed RDAP structures

## Architecture

### Rust Workspace (rdapify-rust)

The Rust workspace has been restructured from a monolithic crate into **16 crates** with explicit feature flags:

| Crate | Purpose | Default |
|-------|---------|---------|
| `rdap-types` | RDAP response types | always |
| `rdap-security` | SSRF + network validation | always |
| `rdap-bootstrap` | IANA service discovery | always |
| `rdap-core` | HTTP fetcher + normalizer | always |
| `rdapify-client` | High-level client | always |
| `rdap-config` | `rdapify.toml` config system | optional |
| `rdap-logging` | Structured logging | optional |
| `rdap-cache` | In-memory response cache | `memory-cache` feature |
| `rdap-stream` | Async streaming queries | `stream` feature |
| `rdap-batch` | Batch query execution | `batch` feature |
| `rdap-rate-limit` | Per-registry rate limiting | `rate-limit` feature |
| `rdap-service` | HTTP service runtime | `service` feature |
| `rdap-sqlite` | SQLite backend (skeleton) | `sqlite` feature |
| `rdap-postgres` | PostgreSQL backend (skeleton) | `postgres` feature |
| `rdap-mysql` | MySQL backend (skeleton) | `mysql` feature |
| `rdapify` | Public API facade | — |

### Feature Flags

```toml
# Minimal (core only, smallest binary)
rdapify = { version = "0.4.0", default-features = false }

# Default (memory cache + streaming)
rdapify = "0.4.0"

# Full open-source build
rdapify = { version = "0.4.0", features = ["full"] }

# With HTTP service
rdapify = { version = "0.4.0", features = ["service"] }
```

## License System (@rdapify/pro)

The license activation system has been upgraded from HMAC-SHA256 to **Ed25519**:

- Activation tokens are Ed25519-signed by the license server
- Machine fingerprinting binds the license to a specific device
- In-process session registry prevents direct class instantiation
- Time rollback detection prevents clock manipulation attacks

See [License Activation](../pro/license-activation.md).

## Breaking Changes

None for `rdapify` (npm) or `rdapify` (Rust) public API.

**@rdapify/pro only:** Feature class constructors (`Reporter`, `HistoryTracker`, etc.) now require a `sessionToken` from `ProPlugin.activate()`. Direct construction is no longer supported. Usage through `ProPlugin` is unaffected.
