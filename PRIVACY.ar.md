# سياسة الخصوصية

**تاريخ السريان**: 23 مارس 2026
**الإصدار**: 2.0

## الملخص التنفيذي

RDAPify هي **مكتبة جانب العميل** تعمل وفقاً لـ **عمارة الخصوصية من التصميم**. لا نجمع أو نخزّن أو ننقل بيانات شخصية.
تحافظ تطبيقاتك على السيطرة الكاملة على معالجة البيانات، وتبقى أنت مراقب البيانات لأغراض GDPR/CCPA.

**نقطة أساسية**: RDAPify لا ترسل البيانات إلى خوادرنا. تستعلم المكتبة فقط عن أنظمة RDAP العامة وتخزّن النتائج محلياً.

## مبادئ الخصوصية الأساسية

1. **تقليل البيانات**: معالجة البيانات الضرورية فقط للاستعلامات RDAP
2. **الخصوصية من التصميم**: تسريح البيانات الشخصية مفعّل افتراضياً (بدون الحاجة للموافقة)
3. **الشفافية**: توثيق واضح لجميع تدفقات البيانات
4. **السيطرة**: تطبيقاتك تسيطر على جميع الاحتفاظ والاستخدام
5. **بدون تلمتري**: بدون إحصائيات استخدام أو تتبع افتراضياً
6. **مفتوح المصدر**: شفافية كود كاملة للتدقيق الأمني

## كيف تعالج RDAPify البيانات

### ما لا نفعله

- ❌ جمع أو إرسال إحصائيات الاستخدام إلى أي خوادم
- ❌ نقل استعلامات RDAP إلى خوادرنا
- ❌ تضمين كود تتبع مخفي
- ❌ الاحتفاظ بنتائج الاستعلام عبر الجلسات (افتراضياً)
- ❌ بيع أو مشاركة البيانات مع أطراف ثالثة
- ❌ تتبع الاستعلامات الفردية
- ❌ تضمين التلمتري افتراضياً (اختياري فقط إن أُضيف صراحةً)

### ما نفعله

- ✅ الاستعلام عن أنظمة RDAP العامة (حسب توجيه تطبيقك)
- ✅ تخزين الاستجابات محلياً في ذاكرة العملية (قابل للتكوين)
- ✅ تسريح معلومات التعريف الشخصية (PII) افتراضياً
- ✅ توفير خيارات التحكم والتكوين للخصوصية
- ✅ دعم ميزات امتثال GDPR/CCPA
- ✅ احتفاظ جميع المعالجة ضمن البنية الأساسية الخاصة بك

## رسم تدفق البيانات

```
تطبيقك
      ↓
RDAPify (مكتبة جانب العميل)
      ├─ الإدخال: النطاق/IP المراد البحث عنه
      ├─ الخطوة 1: التحقق من صحة الإدخال
      ├─ الخطوة 2: فحص الذاكرة المؤقتة المحلية (في الذاكرة فقط)
      │         لا توجد حركة مرور شبكية هنا
      ├─ الخطوة 3: اكتشاف Bootstrap (سجل IANA فقط)
      │         الاستعلام: https://data.iana.org/rdap/dns.json
      │         الاستجابة: عناوين خوادم موثوقة
      ├─ الخطوة 4: فحوصات الأمان (التحقق من SSRF و عدم السماح بـ IPs الخاصة)
      ├─ الخطوة 5: الجلب من خادم RDAP الموثوق
      │         مثال: https://rdap.verisign.com/com/v1/domain/example.com
      ├─ الخطوة 6: معايرة الاستجابة (توحيد أسماء الحقول)
      ├─ الخطوة 7: تسريح البيانات الشخصية (إزالة البريد و الهاتف و العناوين)
      ├─ الخطوة 8: تخزين النتيجة المعايرة مؤقتاً (محلياً و في الذاكرة)
      └─ الإخراج: بيانات RDAP المسروقة
      ↓
تطبيقك
(أنت تسيطر على الاحتفاظ و التسجيل و التخزين)
```

### ما البيانات التي تغادر البنية الأساسية الخاصة بك

**فقط هذه الطلبات تغادر نظامك:**

1. **طلبات استعلام RDAP** (إلى خوادم RDAP العامة)
   - النطاق: `https://rdap.{registrar}/domain/{query}`
   - IP: `https://rdap.{rir}/ip/{query}`
   - ASN: `https://rdap.{rir}/autnum/{query}`
   - تحتوي على: الاستعلام فقط (نطاق/IP/ASN) الذي بحثت عنه

2. **اكتشاف Bootstrap** (إلى سجل IANA)
   - الرابط: `https://data.iana.org/rdap/dns.json` (عام و مخزّن مؤقتاً)
   - التكرار: مرة واحدة لكل دورة bootstrap (قابل للتكوين)
   - البيانات: لا شيء (طلب HTTPS GET)

**ما لا يغادر البنية الأساسية الخاصة بك:**
- ❌ كود تطبيقك أو بيئته
- ❌ ملفات تعريف الارتباط أو المصادقة
- ❌ البيانات الوصفية للاستعلامات (الطوابع الزمنية و الأرقام)
- ❌ إعدادات أو تكوين التطبيق
- ❌ أي معلومات شخصية عن مستخدميك

## تفاصيل تسريح البيانات الشخصية

### مستويات التسريح الافتراضية

**مستوى الخصوصية: "معياري" (افتراضي)**
```
الحقول المسروقة:
  ✓ contact.email
  ✓ contact.phone
  ✓ contact.fax
  ✓ registrant.name
  ✓ registrant.email
  ✓ registrant.address (كاملة)
  ✓ administrator.email
  ✓ technical.email
  ✓ tech.phone
```

**مستوى الخصوصية: "صارم"**
```
الحقول المسروقة (تتضمن ما سبق زائد):
  ✓ contact.name (جميع الاختلافات)
  ✓ registrar.contactEmail
  ✓ nameserver.email
  ✓ administrative.email
  ✓ ipContact.email
```

**مستوى الخصوصية: "متسامح"**
```
الحقول المسروقة (الأدنى):
  ✓ contact.email (البريد فقط)
  ✓ registrant.email (البريد فقط)
  ✓ personal.email (البريد فقط)
```

### أمثلة التكوين

```typescript
// افتراضي (خصوصية معيارية)
const client = new RDAPClient();
// يسرح البيانات الشخصية تلقائياً

// خصوصية صارمة
const client = new RDAPClient({
  privacy: { level: 'strict' }
});
// يسرح جميع معرّفات الأشخاص

// تسريح مخصص
const client = new RDAPClient({
  privacy: {
    level: 'custom',
    redactFields: ['email', 'phone'],
    preserveFields: ['organization']
  }
});

// تعطيل التسريح (غير موصى به)
const client = new RDAPClient({
  privacy: { enabled: false }
});
// تحذير: مخاطر GDPR/CCPA إذا فعلت هذا
```

## خصوصية وحفظ البيانات المؤقتة

### سلوك الذاكرة المؤقتة

```typescript
// الذاكرة المؤقتة في الذاكرة (L1)
- مخزنة في ذاكرة العملية فقط
- تُمسح عند خروج العملية
- تبقى عند انقطاع الطلب
- TTL قابل للتكوين (افتراضي: 1 ساعة)
- الحجم الافتراضي: 1000 إدخال

// الذاكرة المؤقتة Redis (L2) - اختيارية
- مستمرة عبر إعادات تشغيل العملية
- احتفاظ قابل للتكوين (افتراضي: 24 ساعة)
- موزعة عبر العمليات إذا كنت تستخدم Redis
- أنت تسيطر على وصول Redis والنسخ الاحتياطية
```

### تنظيف الذاكرة المؤقتة (حق الحذف GDPR)

```typescript
// امسح جميع البيانات المخزنة مؤقتاً
await client.cache.clear();

// امسح إدخال محدد
await client.cache.delete('domain:example.com');

// امسح حسب النمط
await client.cache.deletePattern('domain:*');

// التنظيف التلقائي مع TTL
// (افتراضي: 1 ساعة و قابل للتكوين)
```

### تكوين الذاكرة المؤقتة للامتثال

```typescript
// الحد الأدنى من الـ caching (موجه للامتثال)
const client = new RDAPClient({
  cache: {
    enabled: true,
    ttl: 300,          // 5 دقائق
    maxSize: 100       // ذاكرة صغيرة
  }
});

// بدون caching (الأكثر حماية للخصوصية)
const client = new RDAPClient({
  cache: {
    enabled: false
  }
});
```

## معالجة البيانات الجهات الثالثة

### أنظمة RDAP (مصادر البيانات العامة)

يستعلم RDAPify عن هذه أنظمة RDAP العامة التي يتم تشغيلها بشكل مستقل:

**سجل النطاق**
- `.com` و `.net`: Verisign (https://rdap.verisign.com)
- نطاقات عامة: مشغلو سجلات ICANN
- نطاقات البلد: سجلات خاصة ببلد معين

**سجل IP**
- ARIN (الأمريكتان): https://rdap.arin.net
- RIPE (أوروبا): https://rdap.db.ripe.net
- APNIC (آسيا-المحيط الهادئ): https://rdap.apnic.net
- LACNIC (أمريكا اللاتينية): https://rdap.lacnic.net
- AFRINIC (أفريقيا): https://rdap.afrinic.net

**آثار خصوصية البيانات:**
- هذه سجلات عامة (بيانات WHOIS)
- أنت مسؤول عن خصوصية المسجل في وقت التسجيل
- RDAPify يسترجع فقط ما هو متاح للجمهور بالفعل
- يتم التسريح على جانب العميل (السجلات ترى الاستعلامات لكن ليس التسريح الخاص بنا)

### خوادرنا

**الوضوح**: RDAPify لا ترسل البيانات إلى خوادم rdapify.com.

لكن إذا استخدمت ميزات @rdapify/pro:
- التحقق من الترخيص (مشفر)
- إخطارات Webhook (إذا كانت مكونة)
- التحليلات (اختيارية فقط)

هذه الميزات الاختيارية:
- مشفرة دائماً أثناء النقل (TLS)
- تُنقل فقط إذا كانت مفعّلة صراحةً
- موثقة في وثائق @rdapify/pro

### التلمتري الاختياري

**الافتراضي**: معطّل
**كيفية التفعيل** (إن لزم الأمر):
```typescript
const client = new RDAPClient({
  telemetry: {
    enabled: true,
    endpoint: 'https://your-analytics-server.com'
  }
});
```

**ما يتم جمعه** (إذا كان مفعّلاً):
- أنواع الاستعلام (نطاق/IP/ASN - وليس القيم الفعلية)
- أوقات الاستجابة
- معدلات إصابة الذاكرة المؤقتة
- رموز الخطأ (وليس تفاصيل الخطأ)
- إصدار المكتبة
- وقت التشغيل (إصدار Node.js)

**ما لا يتم جمعه**:
- ❌ أسماء نطاقات فعلية أو عناوين IP أو قيم استعلام
- ❌ معلومات شخصية
- ❌ معرّفات التطبيق
- ❌ عناوين IP للمستخدم
- ❌ رؤوس الطلب (باستثناء رؤوس HTTP القياسية)

## الامتثال لـ GDPR

توفر RDAPify أدوات لدعم متطلبات المادة 32 من GDPR (الأمان):

### الحق في الوصول
- يمكن لتطبيقك الوصول المباشر إلى جميع البيانات المخزنة مؤقتاً
- `await client.cache.getAll()` تعيد جميع الإدخالات
- لا توجد بيانات مخزنة على خوادم RDAPify

### الحق في الترميم
- يتم جلب البيانات بشكل جديد (لا نحتفظ بنسخ قديمة)
- يمكن تحديث البيانات المخزنة مؤقتاً
- `await client.cache.delete('key')` تفرض جلب جديد

### حق الحذف
- امسح الذاكرة المؤقتة: `await client.cache.clear()`
- التسريح يزيل البيانات الشخصية فوراً
- لا توجد بيانات محفوظة بعد انتهاء TTL للذاكرة المؤقتة
- الامتثال للمادة 17 من GDPR: حذف جانب العميل فقط

### الحق في قابلية نقل البيانات
- جميع البيانات بصيغة RDAP JSON قياسية
- يمكن تصديرها من الذاكرة المؤقتة
- لا توجد صيغ مملوكة

### الأساس القانوني
كمراقب بيانات، يجب عليك تأسيس أساس قانوني:
- **المصلحة الشرعية** (الأكثر شيوعاً للبحث عن النطاق)
- **العقد** (إذا كان المسجل عميلاً)
- **الالتزام القانوني** (الامتثال/منع الاحتيال)
- **الموافقة** (إذا كنت قد حصلت عليها)

**دور RDAPify**: معالج بيانات يدعم امتثالك

## الامتثال لـ CCPA

تدعم RDAPify حقوق المستهلك في CCPA:

### متطلبات الإفصاح
- RDAPify لا تجمع "معلومات شخصية" وفقاً لـ CCPA
- المكتبة شفافة حول البيانات التي يتم الوصول إليها
- الذاكرة المؤقتة تحت سيطرة المستخدم

### تطبيق حقوق المستهلك
```typescript
// 1. الحق في المعرفة - الوصول إلى البيانات المخزنة مؤقتاً
const cachedData = await client.cache.getAll();

// 2. حق الحذف - امسح البيانات
await client.cache.clear();

// 3. حق الاختيار - عطّل التلمتري (افتراضي)
// التلمتري معطّل افتراضياً و لا يحتاج إلى اختيار

// 4. حق عدم التمييز
// RDAPify لا تميز بناءً على خيارات الخصوصية
```

### معلومات حساسة شخصياً
- عناوين البريد الإلكتروني وأرقام الهاتف مسروقة افتراضياً
- إذا كنت بحاجة إلى معالجتها، يجب عليك التعامل على مستوى التطبيق
- تنطبق عقوبات CCPA على التطبيق الخاص بك وليس RDAPify

## الاحتفاظ البيانات

### سياسات الاحتفاظ

**بيانات الذاكرة المؤقتة**
- افتراضي TTL: 1 ساعة
- إدخالات ذاكرة مؤقتة قصوى: 1000
- الاحتفاظ بعد الذاكرة المؤقتة: 0 (مسح عند خروج العملية)

**ذاكرة اكتشاف Bootstrap**
- افتراضي TTL: 30 يوماً
- محفوظة محلياً في مجلد العملية
- يمكن مسحها يدوياً

**السجلات**
- السجلات القياسية للتطبيق غير محفوظة بواسطة RDAPify
- تسجيل Audit (إذا كان مفعّلاً) هو مسؤولية التطبيق
- ينصح باحتفاظ السجل: 7-90 يوماً حسب احتياجات الامتثال

**بدون احتفاظ جانب الخادم**
- خوادم RDAPify لا تخزن بيانات الاستعلام
- لا قواعد بيانات طويلة الأجل للعمليات البحثية
- لا جمع البيانات التاريخية

## اعتبارات الأمان

### التشفير

**أثناء النقل:**
- جميع استعلامات RDAP تستخدم HTTPS (TLS 1.2+)
- دعم تثبيت الشهادة (قابل للتكوين)
- ينصح بـ preloading HSTS

**في الحالة الساكنة:**
- بيانات الذاكرة المؤقتة محفوظة في ذاكرة العملية (غير مشفرة)
- إذا كانت حساسة، طبّق التشفير على مستوى التطبيق
- يمكن أن تستخدم ذاكرة Redis وحدة التشفير

### التحكم في الوصول

تطبيقك يسيطر على:
- من يمكنه إنشاء نسخة RDAPClient
- ما الاستعلامات المسموحة
- أذونات الوصول للذاكرة المؤقتة
- مستويات التسريح

RDAPify يفرض:
- حماية SSRF (لا وصول IP داخلي)
- التحقق من المدخلات (لا حقن)
- HTTPS فقط (بدون نصوص واضحة)
