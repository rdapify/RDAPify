# سياسة الأمان — rdapify (Rust)

تغطّي هذه الوثيقة **كريت `rdapify`** على [crates.io](https://crates.io/crates/rdapify)، وربطها بـ Node.js عبر **`rdapify-nd`**، وربطها بـ Python عبر **`rdapify-py`**.

للاطلاع على سياسة أمان حزمة `rdapify` على npm (TypeScript)، انظر [rdapify/SECURITY.md](https://github.com/rdapify/rdapify/blob/main/SECURITY.md).

## 3. نطاق الأمان

| المكوّن | ما نحميه منه |
|---------|-------------|
| **حماية SSRF** | تجاوز عناوين IP الخاصة/المحلية؛ تهريب البروتوكول |
| **التحقق من المدخلات** | الحقن عبر نطاقات أو عناوين IP أو أرقام ASN مُشوَّهة |
| **تحليل استجابات RDAP** | JSON مُشوَّه يُسبب panic؛ هيكل استجابة غير متوقع |
| **`rdapify-nd` (رابط N-API)** | Panic عبر حدود FFI بين Rust وNode.js |
| **`rdapify-py` (رابط PyO3)** | Panic عبر حدود FFI بين Rust وPython |
| **TLS (`rustls`)** | التحقق من الشهادات؛ اختيار مجموعات التشفير |
| **الأداة CLI** | حقن الأوامر عبر المدخلات؛ اجتياز المسارات |
| **سلسلة التبعيات** | كريت خبيث مُحقن عبر `Cargo.lock` |

## 4. الممارسات الأمنية

### `#![forbid(unsafe_code)]`

تفرض جميع الكريتات في هذا المستودع `#![forbid(unsafe_code)]` على مستوى جذر الكريت. يرفض مُترجم Rust أي كتلة `unsafe` في وقت الترجمة، مما يُلغي فئة ثغرات سلامة الذاكرة.

### بدون OpenSSL — rustls فقط

تُعالَج جميع طبقات TLS بواسطة [`rustls`](https://github.com/rustls/rustls)، وهي تطبيق TLS خالص بلغة Rust. لا يوجد أي تبعية لـ `openssl` أو `native-tls`، مما يُلغي ثغرات المستوى C المرتبطة بـ OpenSSL.

### حماية SSRF

تطبيق Rust لحماية SSRF يعكس بدقة تطبيق TypeScript: تُحجب نطاقات RFC 1918 الخاصة، والحلقة المحلية، والمحلية للوصل، والبث المتعدد والمحجوزة قبل أي دقة DNS أو اتصال صادر.

### تدقيق التبعيات

يعمل `cargo audit` عند كل دفع إلى CI عبر GitHub Actions. يفشل الـ workflow إذا كانت أي تبعية لديها تنبيه منشور.

### بيئة تشغيل Tokio غير المتزامنة

كل الإدخال/الإخراج غير المتزامن يستخدم بيئة تشغيل [Tokio](https://tokio.rs/) مع نموذج async/await الآمن للذاكرة في Rust.

### سلامة حدود FFI

تُلتقط حالات Panic في Rust عند حدود FFI وتُحوَّل إلى أخطاء JavaScript (`rdapify-nd`) أو استثناءات Python (`rdapify-py`) قبل عبور حدود اللغة.

## 5. تدوير المفاتيح والإبطال

لا تُدير مكتبة Rust الأساسية `rdapify` وروابطها مفاتيح تشفير ولا توزّعها.

للاطلاع على تدوير مفاتيح الترخيص وإبطالها في `@rdapify/pro`، تواصل مع security@rdapify.com.

## 6. سياسة الإفصاح

تتّبع rdapify-rust الإفصاح المنسق المسؤول:

1. **المُبلِّغ يُرسل** الثغرة عبر GitHub Advisories أو البريد الإلكتروني
2. **إقرار الاستلام** خلال 48 ساعة (4 ساعات لـ emergency@rdapify.com)
3. **تقييم الخطورة** خلال 7 أيام
4. **تطوير الإصلاح** في فرع خاص
5. **نشر إصدار الأمان** — إصدار patch مع قسم `### Security` في CHANGELOG
6. **نشر التنبيه** على GitHub Security Advisories؛ يُطلب CVE إن انطبق
7. **إشادة بالمُبلِّغ** بالاسم أو المعرّف (ما لم يُطلب إخفاء الهوية)

**الحد الأقصى للحظر: 90 يومًا** من تاريخ الإبلاغ الأولي.

**جداول الإصلاح حسب الخطورة:**

| الخطورة | أمثلة | الهدف |
|---------|-------|-------|
| حرج | تجاوز SSRF، تنفيذ كود عن بُعد، عدم سلامة الذاكرة | 7 أيام |
| عالٍ | DoS عبر استجابة مُشوَّهة، انتشار Panic عبر FFI | 14 يومًا |
| متوسط | كشف معلومات، تراجع TLS | 30 يومًا |
| منخفض | توثيق بتداعيات أمنية | الإصدار التالي |

## 7. تحديثات الأمان

تُطلَق إصلاحات الأمان كـ **إصدارات patch** للكريت المتأثر.

قد تُؤدي الثغرات الحرجة إلى **إصدار طارئ** خارج جدول الإصدار العادي.

كل إصدار أمني يتضمّن:
- قسم `### Security` في [`CHANGELOG.md`](CHANGELOG.md)
- تنبيه منشور على [GitHub Security Advisories](https://github.com/rdapify/rdapify-rs/security/advisories)
- تنبيه في [قاعدة بيانات RustSec Advisory](https://rustsec.org/) للمشكلات المؤهلة لـ CVE

**للاشتراك في إشعارات الأمان:**
- GitHub: راقب هذا المستودع ← "Security alerts only"
- Rust: شغّل `cargo audit` في مشروعك بعد تحديث التبعيات
- Python: شغّل `pip-audit` إذا كنت تستخدم `rdapify-py`
